# 正规发布验收清单

本文用于把 LSYL Tunnel 从“可运行工具”收敛成“可正式交付软件”。重点不是“绕过杀软”，而是让安装器、Windows 服务、网络行为和交付材料都足够透明、可解释、可审计，降低被识别为木马、后门或未知常驻程序的概率。

> 服务端 GUI 正在并行调整中。本文不跟踪具体页面改版过程，只保留发布前必须满足的验收边界。

## 1. 发布结论分级

### 阻断发布

正式对外发布前，出现以下任一问题，不建议发布：

- `go test ./...` 未通过。
- `deploy\windows\test\check-text.ps1` 未通过，存在乱码、破损 HTML 或非 UTF-8 文本。
- 服务端或客户端 exe 缺少版本信息、产品名、公司名、版权信息或图标。
- 服务端 Windows 服务缺少固定服务名、显示名、描述、卸载路径或日志位置说明。
- 客户端安装包重新引入客户端 Windows 服务、客户端服务注册或客户端服务残留清理逻辑。
- 服务端 GUI 改版合并后，页面保存结果与实际 `server.yaml` 不一致。
- 服务端转发端口配置与业务边界不一致：反向端口不唯一、反向端口非本机回环、正向目标不可诊断。
- 安装器缺少签名，且发布对象不是内部测试人员。
- 使用压缩壳、混淆壳、隐藏释放、自删除、随机服务名、随机进程名等容易触发杀软风控的发布方式。

### 可以内部测试

满足以下条件，可以给内部测试人员使用：

- 客户端和服务端核心流程可跑通。
- 安装、升级、卸载基本可验证。
- 使用本机自签代码签名或测试证书签名。
- 已知杀软误报有记录，且能说明误报文件、签名状态、SHA256 和触发动作。
- 已知问题写入发布说明，且不会造成数据丢失或安全边界误解。

### 可以正式交付

满足以下条件，才作为正式交付版本：

- 阻断发布项全部清零。
- 使用正式代码签名证书签名安装器和落地 exe。
- 发布包附带版本号、构建时间、文件 SHA256、配置说明和安装说明。
- 客户端与服务端安装路径、服务名、程序名、卸载项和图标统一为 `LSYL Tunnel`。
- 服务端服务注册、网络监听、证书生成、日志目录和卸载保留目录均有明确文档说明。
- 对主流杀软或系统安全提示做过安装、启动、卸载验证，并保留结果截图或记录。

## 2. 功能正确性

- 客户端登录成功后进入托盘值守，关闭窗口不断开连接。
- 客户端未连接时关闭窗口直接退出。
- 客户端连接失败必须给出可理解提示，不能长期停留在“连接中”。
- 客户端不显示高级端口映射配置，端口映射由管理员预置。
- 服务端 GUI 改版完成后，“用户认证”页只管理用户、禁用状态、重置密码和绑定端口展示。
- 服务端 GUI 改版完成后，“转发端口”页按已有用户分配端口放通关系。
- 服务端 GUI 改版完成后，保存转发端口必须正确写回对应规则的 `allowed_users`。
- 服务端 GUI 改版完成后，切换页签自动刷新配置和状态，未保存修改应恢复为配置值。

## 3. 转发业务边界

- 正向转发 `client_to_server`：GUI 保存时预检服务端目标端口，服务端启动后审计并记录目标状态。
- 正向目标不可访问时，服务主入口不应被拖垮；实际连接失败和运行详情必须给出直接原因。
- 反向转发 `server_to_client`：服务端端口只允许配置为本机回环地址。
- 反向端口在服务端配置中必须全局唯一。
- 反向端口必须唯一归属一个启用用户，不能多个用户复用同一个反向端口。
- 服务端启动时只加载和审计反向端口；客户端激活时才尝试监听该端口。
- 反向端口被占用时属于软失败，客户端必须给出“服务端被动端口不可用”提示、记录错误并自动重试。
- 客户端异常离线时，服务端通过反向控制心跳释放客户端激活状态。

## 4. 安全边界

- 客户端身份认证只使用账号密码或服务端密封的短期登录凭据，不使用客户端证书。
- TLS 用于保护账号密码和业务流量，同时校验服务端身份。
- 客户端安装包只携带 `server.crt`，不得携带 `server.key`、登录密封私钥或运行数据。
- 服务端保存密码哈希，生产环境不得使用 `plain:` 密码。
- 客户端不长期保存明文密码。
- 客户端保存的密封凭据过期后必须重新输入密码。
- 服务端失败封禁 IP 持久化到 `data`，服务重启后未过期封禁仍生效。
- 服务端监控接口默认只监听本机地址。

## 5. Windows 服务正规性

- 服务端服务名固定为 `LSYLTunnelServer`，不得随机生成、不得伪装系统服务。
- 服务端服务显示名固定为 `LSYL Tunnel Server`。
- 服务端服务描述必须说明用途，例如“LSYL Tunnel encrypted tunnel server service”。
- 服务端服务启动类型默认手动；只有安装向导明确让管理员选择时，才允许设置为开机自启动。
- 服务端服务程序路径必须位于安装目录 `bin` 下，不能释放到临时目录、用户缓存目录或系统目录。
- 服务端服务启动参数必须明确包含配置路径和日志路径，便于管理员审计。
- 服务端服务日志按天写入 `logs`，不得只写入隐藏目录。
- 服务端注册、启动、停止、删除服务失败时，必须输出直接原因，不能吞错。
- 安装、升级、卸载只通过服务控制管理器正规停止服务，不使用强制 `taskkill /F` 清理服务进程。
- 客户端不得注册服务；客户端常驻只通过用户态进程和托盘显式呈现。
- 客户端安装、升级、卸载只请求 GUI `/quit` 温和退出；失败时提示用户从托盘退出，不默认强制结束进程。

## 6. 安装器与系统表现

- 客户端安装器不注册 Windows 服务。
- 服务端安装器注册 `LSYLTunnelServer`，默认启动类型为手动，并提供明确的开机自启动确认项。
- 安装器不落地 `.ps1/.cmd` 安装/卸载脚本。
- 客户端卸载不留下客户端服务、重复托盘图标或残留运行进程。
- 客户端安装器不得使用 `taskkill /F` 强制清理 GUI 进程。
- 客户端安装器在退出请求失败或安装中断时必须清理本次临时 helper/回滚目录，并尽量恢复原配置和证书。
- 服务端卸载删除服务注册，默认保留 `conf`、`certs`、`data`、`logs`。
- 安装器名称、发布者、图标、卸载项、快捷方式和程序标题一致。
- 安装器运行时需要管理员权限的动作必须有明确说明。
- 安装或升级不误删非安装目录。
- 安装器不得静默关闭安全软件、修改系统安全策略、添加隐藏计划任务或隐藏自启动项。
- 安装器不得把二进制释放到随机目录后执行。
- 卸载器必须停止并删除服务注册，但默认保留配置、证书、数据和日志。

## 7. 代码签名与发布信誉

- 正式发布必须使用正式代码签名证书，不使用本机自签证书。
- 签名对象包含分发包内 exe 和最终安装器。
- 签名必须带时间戳。
- 客户端 GUI、服务端 GUI、服务端服务 exe、命令行工具和安装器都必须签名。
- 所有 exe 必须带一致的版本资源：产品名、文件说明、公司名、版权、版本号和图标。
- 同一产品尽量长期使用同一正式证书发布，积累信誉。
- 发布前执行：

```cmd
cmd /c release.cmd /verify-only
```

- 发布后抽查安装器“数字签名”页。
- 如遇杀软误报，保留样本哈希、签名状态、安装器版本和误报截图，向对应厂商提交误报申诉。

## 8. 杀软误报处理材料

出现“木马”“后门”“可疑服务”“未知发布者”等拦截时，不做规避处理，按正规软件申诉材料准备：

- 文件名、版本号、构建时间、SHA256。
- 文件数字签名截图或 `signtool verify` 输出。
- 安装器和落地 exe 的路径列表。
- Windows 服务名、显示名、描述、启动类型、服务命令行。
- 安装动作说明：写入目录、生成证书、注册服务、创建快捷方式、保留目录。
- 网络行为说明：服务端监听端口、监控地址、客户端连接地址、业务转发用途。
- 日志位置说明：`logs` 目录和按天日志文件。
- 卸载行为说明：删除程序文件和服务注册，默认保留 `conf`、`certs`、`data`、`logs`。
- 误报截图、杀软名称、病毒库版本和触发动作。

## 9. 不做的事

以下行为短期看似可能减少拦截，长期会显著增加木马特征，禁止作为发布策略：

- 不加壳、不混淆、不压缩可执行文件。
- 不随机化文件名、服务名、进程名或安装路径。
- 不绕过 UAC，不隐藏管理员授权动作。
- 不关闭或修改杀软、防火墙、安全中心策略。
- 不用计划任务、注册表 Run 项或隐藏服务做客户端常驻。
- 不在临时目录释放二进制后执行。
- 不自删除安装器、不删除自身日志、不清空系统事件。
- 不把私钥、密码、PFX 或签名证书放入发布包。

## 10. 文档与交付材料

正式交付至少包含：

- 安装器：`LSYL-Tunnel-Client-Setup.exe`、`LSYL-Tunnel-Server-Setup.exe`
- 部署说明：`docs/deployment/windows-deployment-zh.md`
- 安全说明：`docs/system/security-zh.md`
- 签名说明：`docs/release/signing-zh.md`
- 发布清单：本文档
- 发布产物清单：`dist\release-manifest.txt`、`dist\release-manifest.json`
- 误报申诉材料：文件 SHA256、签名状态、服务说明和安装行为说明，可直接从发布产物清单摘取

发布说明中必须写明：

- 软件名称和版本。
- 适用系统。
- 客户端不注册服务。
- 服务端会注册 `LSYLTunnelServer` 服务。
- 服务端证书私钥只留在服务端。
- 客户端只需要 `server.crt`。
- 默认保留目录和卸载行为。

## 11. 当前非 GUI 待整改项

- 需要确认服务端服务 exe、CLI 工具和安装器均被签名，不只签 GUI。
- 服务端核心转发预检应保持“硬配置、软启动”：配置阶段校验反向端口唯一归属，运行阶段由客户端激活并记录端口不可用。
- 当前自签代码签名只适合开发或内部测试，不能作为正式对外发布信誉。
- 仍需要做一次真实安装、升级、卸载、服务注册、服务启动、服务停止和服务卸载验证。
- 需要形成一份“服务为什么不是木马”的交付说明：服务名、用途、端口、日志、卸载和保留目录一目了然。

已纳入自检：

- 所有发布 exe 必须带版本资源、产品名、公司名、文件说明、版本号和图标。
- 发布流程必须输出 `dist\release-manifest.txt/json`，记录构建时间、SHA256、签名状态、版本资源和服务说明。
- 客户端不得出现 Windows 服务注册或客户端服务 exe。
- 客户端安装器不得使用 `taskkill /F` 强制清理 GUI 进程。
- 服务端安装器不得使用 `taskkill /F` 强制清理服务进程。

## 12. 服务端 GUI 改版后验收项

服务端 GUI 改版由页面实现侧推进，合并到发布版本前按以下标准验收：

- 页面和 Go 源码必须为 UTF-8，`check-text.ps1` 不能发现乱码、破损 HTML 或错误转义。
- 页面保存用户、转发端口和安全参数后，落盘配置必须与页面含义一致。
- 转发端口页面必须以用户为中心写回授权，不再出现页面显示已绑定但 `server.yaml` 未生效的情况。
- 服务启动失败时，页面必须直接提示端口、目标、证书、权限或服务状态原因。
- 页面空白处不显示浏览器右键菜单、页面源码或检查菜单。
- 页面切换、最大化、恢复和刷新不能造成输入卡死、频繁重绘或配置被自动覆盖。

## 13. 推荐整改顺序

1. 确认服务端服务注册信息正规：固定服务名、显示名、描述、默认手动启动、开机自启动需确认、日志路径、卸载行为。
2. 配置正式代码签名证书和时间戳服务器，签包内 exe 和最终安装器。
3. 持续完善服务端转发端口预检、客户端反向激活失败提示和运行详情记录。
4. 做一次真实安装、升级、卸载和服务控制验证。
5. 准备误报申诉材料包，记录样本哈希、签名、版本、服务行为和截图。
6. 等服务端 GUI 改版合并后，再统一执行 `go test ./...`、`check-text.ps1` 和页面验收。
