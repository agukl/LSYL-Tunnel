# 运行详情第二期设计

## 当前第一期边界

第一期页面只做“能快速看懂当前发生了什么”：

- 封禁 IP：读取服务端持久化状态，显示未过期的封禁来源 IP、剩余时间和自动解封时间。
- 失败诊断：从服务端内存滑窗事件中过滤认证失败、策略拒绝、目标失败、反向端口无客户端等事件。
- 最近事件：展示最近请求、登录探测、正向连接、反向控制连接和反向入口事件。
- 日志兜底：运行事件会同步写入每天一份的服务日志，页面滑窗只负责近期排查。

第一期不做主动解封、不做长期统计、不做连接级实时拓扑。

## 第二期目标

第二期要把“运行详情”从近期事件列表升级为运维诊断页：

- 活跃连接：显示当前正在转发的连接，包括用户、方向、转发名称、来源 IP、目标地址、开始时间、持续时间、上/下行流量。
- 用户维度统计：显示每个用户的认证成功、认证失败、策略拒绝、活跃连接、累计流量。
- 转发端口维度统计：显示每条转发规则的连接次数、目标失败次数、当前活跃连接、累计流量。
- 封禁管理：支持管理员手动解除单个 IP 封禁，并记录操作日志。
- 目标诊断：正向代理显示服务端目标不可达原因；反向代理显示反向端口是否已被客户端激活。
- 页面过滤：按用户、端口、事件类型、结果筛选近期事件。

## 后端接口建议

新增管理台内部接口：

```text
GET /api/runtime/details
```

返回建议：

```json
{
  "active_connections": [],
  "user_stats": [],
  "forward_stats": [],
  "blocked_ips": [],
  "recent_events": []
}
```

新增封禁管理接口：

```text
POST /api/runtime/unblock
```

请求：

```json
{ "ip": "192.0.2.10" }
```

接口边界：

- 仅本机管理台使用，不暴露到隧道公网监听端口。
- 不返回密码、密码哈希、客户端保存凭据密文。
- 不把完整业务数据包写入日志，只记录连接元信息和结果。

## 实现步骤

1. 在服务端连接生命周期里分配连接 ID，并在连接开始、关闭时维护 `active_connections`。
2. 将现有计数器拆分到用户维度和转发规则维度。
3. 给封禁状态增加 `Unblock(ip)` 方法，写回 `runtime.state_file`。
4. GUI 增加 `/api/runtime/details` 和 `/api/runtime/unblock`，统一从服务端监控接口和持久化状态聚合数据。
5. 前端在“运行详情”页增加筛选器、活跃连接表、用户统计表、转发统计表和解除封禁按钮。
6. 自检脚本补充认证失败封禁、手动解封、正向目标失败、反向无客户端四类场景。

## 验收标准

- 管理员能在 30 秒内判断：是账号密码错、用户被禁用、端口未授权、目标服务不可达，还是反向端口没有客户端激活。
- 服务停止时仍能看到未过期封禁 IP。
- 服务运行时能看到活跃连接和最近事件。
- 手动解除封禁会立即生效，并写入操作日志。
- 页面不会展示敏感凭据内容。
